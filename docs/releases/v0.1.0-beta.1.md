# TwinTidy v0.1.0-beta.1

- Release type: public prerelease candidate
- Publisher: Kayilan Inc
- License: MIT
- Platform: Windows x64 and Windows ARM64

TwinTidy `v0.1.0-beta.1` is the first public prerelease candidate of the Windows-native exact-duplicate review application. It is intended for scanning, verification, and review. It does not change or remove files.

## Important safety limitation

Cleanup is disabled in this release. Windows' documented Shell recycle operation is path-based and cannot guarantee that the object recycled is the same file identity TwinTidy verified. The production adapter therefore stops before any destructive native call. Checked rows are cleanup-planning selections only: no file is recycled, deleted, renamed, or otherwise modified.

Do not use this prerelease if your requirement is automated duplicate removal. Continue to manage any files manually after independently reviewing them.

## Highlights

- Native Windows desktop interface built with Go and Walk, with no auxiliary runtime or background service.
- Separate x64 and ARM64 builds with matching manifests, product icon, version information, and per-monitor DPI support.
- Staged duplicate discovery using exact size, boundary xxHash, and full streaming SHA-256 confirmation.
- Stable Windows file identities, hard-link and alternate-stream protection, reparse-safe traversal, and cancellable hashing.
- File-type focus filters, group review, keep-newest and keep-oldest planning helpers, Windows previews, and local diagnostics.
- Generation-scoped UI work so cancelled or stale callbacks cannot replace newer scan state.
- Fail-closed cleanup capability gating required by the independent safety review.
- Reproducible unsigned builds, signed-byte provenance, SHA-256 checksums, protected signing, and immutable release assets.
- Per-user MSIX installation and Windows-managed update and uninstall, with the portable ZIP retained as a supported alternative.

## Publisher and artifact verification

The protected release job must record the exact public code-signing certificate subject and SHA-256 thumbprint on the GitHub release before this candidate is promoted. Every distributed executable and MSIX must have a valid trusted timestamped signature matching those values. The MSIX manifest `Publisher` must exactly equal the executable signer certificate subject; `Kayilan Inc` is the human-readable publisher name, not a substitute for that complete certificate identity.

Compare each download with `SHA256SUMS.txt`, then verify its signature as described in the installation guide. Do not install an unsigned artifact, a file whose checksum differs, or a package that shows an unexpected or unverified publisher.

## Installation and updates

- The portable ZIP runs from an extracted user directory and is updated manually by extracting a verified newer version into a separate directory.
- The MSIX installs for the current user without elevation and uses Windows for Start menu registration, higher-version updates, and uninstall.
- The prerelease semantic version maps to MSIX version `1.1.0.1`; MSIX adds one to the SemVer major because package major zero is invalid.
- A future stable `0.1.0` would map to `1.1.0.0` and cannot replace this prerelease in place. The protected tag gate rejects that non-increasing transition, so the next public version must map higher.
- Architecture changes and downgrades require uninstalling the current MSIX before installing the intended signed package.

## Known limitations

- Cleanup and disk-space reclamation are unavailable.
- Only Windows x64 and Windows ARM64 are supported; Windows 32-bit builds are not provided.
- Portable packages have no automatic updater.
- Direct MSIX installation requires Windows App Installer or an organization-approved MSIX deployment mechanism and trust in the public signing-certificate chain.
- Native startup, installation lifecycle, and accessibility acceptance must pass on both x64 and ARM64 before publication.

## Publication gate

This file describes a release candidate, not evidence that public artifacts already exist. Promotion requires all default-branch and protected-release checks, clean exact-commit reproducible builds, native x64 and ARM64 smoke results, signed executable and MSIX verification, MSIX install/update/uninstall evidence, final checksums and provenance, and the independent safety review. Artifacts must be published initially as an immutable GitHub prerelease and must never be replaced under the same version.

Security or possible data-loss reports should use [GitHub private vulnerability reporting](https://github.com/Soyuz-Tec/duplicate-file-finder-go/security/advisories/new), not a public issue.
